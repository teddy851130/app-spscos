## 규칙
사용자가 본 현상에 대해 "수정했다"고 보고하기 전에, 내가 먼저 수정 + 한 번 더 검증(실행/DB/UI/Edge Function 현재 배포 버전 확인) 후 보고한다.

**Why:**
- 2026-04-21 Sprint04 Day 2: PR17~PR22 6커밋 "배포 완료" 보고했으나 실제 사용자(Teddy)가 4/22 본 화면에는 Dear/CEO 서명/P.S. URL/Korea 누락/파이프라인 D/E UI 잔존 등 PR17 이전 상태가 그대로. 원인은 (a) Edge Function 재배포 누락 가능성 (b) Pipeline.tsx UI 코드 수정 누락 (c) 프롬프트 코드와 실제 실행 결과의 gap을 DB로 교차 확인하지 않음.
- Teddy 직접 피드백: "왜 같은 말을 계속 반복해야 하는가. 나한테 수정했다고 하기 전에, 네가 수정하고 한 번 더 검토하고 보고해."

**How to apply:**
1. 코드 수정 후 반드시 **실제 동작 결과**로 교차 검증한 뒤에만 "완료" 보고.
2. 검증 경로:
   - **프롬프트/프로세스 변경**: 코드 수정 → Edge Function 재배포 → 실제 DB에서 최근 실행 결과(email_drafts.body_first 등) 확인 → 프롬프트 규칙이 결과에 반영됐는지 대조
   - **UI 변경**: 로컬 dev 서버 or 배포 후 실제 페이지 접속 → 스크린샷/DOM 구조 확인 (브라우저 MCP 또는 사용자 확인 요청)
   - **DB/스키마 변경**: SELECT로 실제 적용 결과 확인
3. 교차 검증 없이 "배포 완료" 표현 금지. 검증 안 했으면 "코드 반영 완료, 실제 동작 검증 대기" 같이 상태 구분.
4. "배포"와 "반영"은 다르다. 코드 커밋·푸시 = 배포가 아니다. Edge Function은 별도 deploy 명령 필요, UI는 Vercel 빌드 성공 + 브라우저 캐시 이슈까지 확인해야 함.

## 관련 사건
- 2026-04-22: PR17/18/19/21/22 코드는 main에 푸시됐으나 generate-draft Edge Function이 실제 PR17 프롬프트로 재배포됐는지 미검증. Pipeline.tsx UI는 D/E 스텝 하드코딩 제거가 PR18에 포함되지 않아 그대로 노출.
